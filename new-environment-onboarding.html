Overview
What is this skill?
An interactive Claude agent skill that collects environment inputs and writes every required file to disk in one session — runtime configs, Kubernetes manifests, CI/CD pipelines, secrets commands, DNS records, and deployment ordering.
💬

Guided Q&A

Collects environment ID, cluster name, service list, DB endpoints, message broker addresses, and service host mappings through targeted questions — no prior context needed.

📁

Writes Files to Disk

Generates all config files and saves them directly to the correct repo paths. No copy-paste, no manual templating, no missed overrides.

🔀

Service-Type Aware

Branches on service deployment type: API-only, worker-only, API+worker, or API+worker+DB. Each path generates the right file subset with the right structure.

🧠

Pitfalls Baked In

Known failure modes from real environment onboardings are encoded as guardrails — wrong placeholders, misconfigured canaries, UUID mismatches all prevented at generation time.

Templates
Service Type Templates
The skill detects the deployment type for each service and generates the correct file set. Four templates cover all cases.
API Only

Stateless API Service

  • Runtime config (DB, cache, messaging)
  • Kubernetes deployment + ingress
  • gRPC + HTTP pipeline
  • Canary analysis config
  • DNS: ingress + internal gRPC records
Worker Only

Async Event Consumer

  • Runtime config (messaging, DB if needed)
  • Kubernetes deployment (no ingress)
  • Worker-only pipeline (no canary HTTP check)
  • commit_txt_host empty, failPipeline=false
  • Correct container name in deployments_config
API + Worker

Hybrid Service

  • Runtime config for both API + consumer
  • Two Kubernetes deployments
  • Pipeline with separate API + worker stages
  • Ingress for API, no ingress for worker
  • Full DNS record set
API + Worker + DB

Full-Stack Service

  • All of API+Worker plus DB migration job
  • DB credentials in secrets
  • Migration pipeline stage before deploy
  • Schema and connection pool config
  • DB port validation (proxy vs direct)
Process
Onboarding Phases
Two phases: environment-level infrastructure, then per-service configuration.
PHASE 1 Environment & Infrastructure Setup
1

Cluster Registration

Register the new cluster with your deployment orchestrator so pipelines can target it. Generates the cluster config entry.

2

Network Infrastructure

Security groups, VPC peering, subnet routing for the new environment. DB proxy / load balancer setup if required by the environment's access pattern.

3

Secrets Bootstrap

Provision DB credentials, message broker credentials, and service tokens into secrets management. Skill generates ready-to-run credstash put commands for all services.

4

DNS Records

Create DNS entries for each service's internal gRPC host, external ingress, and concierge endpoint. Three record types per service, generated in bulk.

PHASE 2 Per-Service Configuration
1

Runtime Config

Service-specific config file: DB connection, message broker addresses, peer service hosts, cache config, feature flags, org/tenant IDs. Auto-wired with environment name, region, and topic names.

2

Kubernetes Manifest

Namespace, replicas, HPA bounds, resource limits, ingress class/host. Critical overrides like grpc_internal_host always set explicitly — never left as a template placeholder.

3

Deployment Pipeline

Canary analysis config with correct lifetime. Pipeline JSON with UUID matching the canary config. Both files generated atomically — UUID mismatch is structurally impossible.

4

Orchestrator Entry

Add service stage to the master deploy orchestrator in the correct dependency order. Pipeline IDs marked for post-sync verification.

PHASE 3 Verify & Deploy
1

Raise PRs

One PR per repo: runtime configs, Kubernetes manifests, deployment pipelines, infrastructure/DNS. All files already correct — review and merge.

2

Post-Sync Verification

After the deployment system syncs pipelines, verify orchestrator stage IDs match the assigned pipeline IDs. Update if needed.

3

Ordered First-Install Deploy

Deploy services in dependency order — services with no upstream dependencies first, consumers and aggregators last. The skill generates this order based on your service graph.

Guardrails
Pitfalls prevented at generation time
These are real failure modes from production environment onboardings. Every one is encoded as a guardrail — the skill makes them structurally impossible to hit.
Critical

Critical placeholder left unsubstituted

Symptom: Service starts cleanly but all calls to peer services fail at runtime with DNS errors or timeouts. The template variable was copied from another environment and never replaced with the real hostname.

Guardrail: Skill writes the actual resolved hostname at generation time. No template variables are left in output files — every value is concrete.
Critical

Canary lifetime = 0 exits instantly

Symptom: Canary analysis stage completes in seconds and reports "insufficient data". The canary was never actually evaluated — it exited before collecting any metrics.

Guardrail: Skill enforces canary_lifetime_mins ≥ 1. Value 0 is rejected at generation time with an explanation.
Critical

UUID mismatch between canary config and pipeline

Symptom: Pipeline fails to resolve the canary config at deploy time — "config not found". Caused by copying files from another service and updating only one of the two UUID references.

Guardrail: Both files are generated in the same call with a single UUID written to both atomically. Mismatch is structurally impossible.
Warning

Worker-only service references wrong container name

Symptom: Pipeline's commit-check stage fails or targets the wrong container for worker-only services. The deployments_config was copied from an API service and still references the api container.

Guardrail: Skill detects worker-only services and sets the correct container name, clears the commit host, and sets failPipeline: false for the commit-check stage.
Warning

Orchestrator pipeline IDs stale after sync

Symptom: Master deploy orchestrator references old or wrong pipeline IDs after the deployment system syncs new pipelines. The orchestrator runs but triggers the wrong pipelines.

Guardrail: Skill marks all orchestrator IDs as "verify post-sync" with clear inline comments. A checklist step is generated as part of the deploy phase.
Critical

DB port unreachable in new environment

Symptom: Service pods crash on startup with a DB connection timeout. The config was copied from an existing environment that routes through a DB proxy on a non-standard port — the proxy isn't set up for the new environment yet.

Guardrail: Skill asks "Is the DB proxy configured for this environment?" and selects the correct port accordingly. Adds a TODO comment if proxy setup is pending.

# Claude will ask:
? Environment ID:           my-cell-prod
? Environment name:         prod
? Cluster name:             my-cluster-blue
? DB host:                  db.internal.myservice.vpc
? DB port (direct/proxy):   5432  # or non-standard proxy port
? Message broker addresses: broker-1.internal:9096, broker-2.internal:9096
? Internal service domain:  int.prod.myservice.com
? Org / tenant ID:          12345
? Services to onboard:      all  # or comma-separated list

# Output: all files written to disk across all repos
# Raise one PR per repo — review and merge
